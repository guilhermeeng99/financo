# Financo — Copilot Instructions

You are assisting in a **Flutter personal finance application** called **Financo**.
Financo is a personal financial manager focused on account reconciliation, expense tracking, and AI-powered data entry via an in-app chat. The goal is to provide a clear, reliable overview of personal finances — income, expenses, balances, and trends — with minimal manual effort.

Generate **high-quality, performant, and maintainable** code that follows the conventions established in this codebase.

### Feature Structure (each feature follows this pattern)

```
feature_name/
├── data/
│   ├── datasources/              # Remote/local data sources
│   ├── models/                   # JSON serialization (fromJson/toJson, fromFirestore)
│   └── repositories/             # Repository implementations
├── domain/
│   ├── entities/                 # Pure Dart classes (no external deps)
│   ├── repositories/             # Abstract repository contracts
│   └── usecases/                 # Single-responsibility use cases
└── presentation/
    ├── bloc/ or cubit/           # State management
    ├── pages/                    # Route-level screens
    └── widgets/                  # Feature-specific widgets
```

### Data Flow

```
UI (Page/Widget) → Bloc/Cubit → UseCase → Repository (abstract) → DataSource (Firebase)
                                              ↕ AppDataCache (in-memory)
```

- Domain entities are plain Dart classes with no external dependencies.
- Models (data layer) contain `fromJson`/`toJson` and `fromFirestore` conversions.
- Repositories are abstract in domain, implemented in data.
- `get_it` for dependency injection (service locator pattern).
- `Either<Failure, T>` from `dartz` for error handling in use cases.
- Repositories check `AppDataCache` before hitting Firebase; store results in cache after fetch.

## State Management — flutter_bloc

- **Bloc** for features with complex event-driven logic (Auth, Dashboard, Transactions, Chat).
- **Cubit** for simpler state (Accounts, Categories, Profile, Startup).
- Every Bloc/Cubit state extends `Equatable`.
- States use sealed classes or enums for status (initial, loading, loaded, error).
- Never emit states with raw strings for errors — use `Failure` objects.

### Singleton Blocs (global, registered as `registerLazySingleton`)

| Bloc/Cubit | Scope | Notes |
|---|---|---|
| `AuthBloc` | Global | Singleton. Auth state drives router redirects. |
| `DashboardBloc` | Global | Singleton. Pre-loaded at startup. Financial summary. |
| `TransactionsBloc` | Global | Singleton. Pre-loaded at startup. Guard: skips fetch if already loaded. |
| `AccountsCubit` | Global | Singleton. Pre-loaded at startup. All user accounts. |
| `CategoriesCubit` | Global | Singleton. Pre-loaded at startup. All categories. |
| `ProfileCubit` | Global | Singleton. Pre-loaded at startup. User data + preferences. |

- Global blocs are provided in `app_widget.dart` via `MultiBlocProvider` with `BlocProvider.value`.
- **Do NOT wrap pages in local `BlocProvider`** for singleton blocs — they are already in the widget tree.
- Use `forceRefresh: true` parameter + dedicated refresh events (e.g. `DashboardRefreshRequested`) for pull-to-refresh.

### Factory Blocs (created per use, registered as `registerFactory`)

| Bloc/Cubit | Scope | Notes |
|---|---|---|
| `ChatBloc` | Per session | Created in router. Manages Gemini conversation. |
| `ReportsCubit` | Per view | Created in router. Fetches data for selected period. |
| `TransactionFormCubit` | Per form | Created locally for add/edit transaction flow. |
| `AccountFormCubit` | Per form | Created locally for add/edit account flow. |

## Error Handling

- Use cases return `Either<Failure, T>`.
- Failures are a sealed class hierarchy: `ServerFailure`, `AuthFailure`, `CacheFailure`, `AiFailure`.
- DataSources throw `ServerException`, `AuthException`, `AiException`.
- Repositories catch exceptions and return `Left(Failure)`.
- Blocs/Cubits handle `Either` and emit appropriate states.

## Chat AI — Google Gemini Integration

### Overview

The Chat feature allows the user to log transactions via natural language. The in-app chat communicates with Google Gemini to extract structured financial data from free-text messages.

### Flow

```
User sends message (e.g. "Gastei 45 reais no almoço hoje")
  → ChatBloc sends message to Gemini with system prompt + conversation history
  → Gemini extracts: { type: expense, amount: 45.00, category: "Alimentação", date: today, description: "Almoço" }
  → ChatBloc emits state with extracted data for user confirmation
  → User confirms or edits → ChatBloc calls CreateTransactionUseCase
  → Transaction saved to Firestore → Success message in chat
```

### System Prompt (Gemini)

The Gemini system prompt instructs the model to:
1. Act as a financial assistant that extracts transaction data from messages.
2. Always respond in Portuguese (pt-BR).
3. Extract: `type`, `amount`, `category`, `date`, `description`, `accountId` (if mentioned).
4. Ask for clarification when data is ambiguous or incomplete.
5. Return structured JSON alongside a friendly confirmation message.
6. Never fabricate data — ask the user if unsure.

### Architecture

```

├── data/
│   ├── datasources/
│   │   ├── chat_remote_datasource.dart    # Firestore (chat history persistence)
│   │   └── gemini_datasource.dart         # Google Generative AI SDK calls
│   ├── models/
│   │   └── chat_message_model.dart
│   └── repositories/
│       └── chat_repository_impl.dart
├── domain/
│   ├── entities/
│   │   └── chat_message_entity.dart
│   ├── repositories/
│   │   └── chat_repository.dart

│       ├── send_message_usecase.dart       # Send message + get Gemini response
│       └── get_chat_history_usecase.dart

    ├── bloc/
    │   ├── chat_bloc.dart
```

- `GeminiDataSource` wraps `google_generative_ai` package.
- API key is stored securely (environment variable / Firebase Remote Config — **never hardcoded**).
- Chat history is persisted to Firestore for context across sessions.
- `TransactionConfirmationCard` renders the extracted data and provides Confirm / Edit / Cancel actions.

## Financial Logic

### Currency Formatting

- All monetary values are stored as `double` in Firestore.
- Display formatting uses `intl` package: `NumberFormat.currency(locale: 'pt_BR', symbol: 'R\$')`.
- Utility function `formatCurrency(double value)` in `core/utils/currency_formatter.dart`.
- Income values are displayed in `AppColors.income` (green).
- Expense values are displayed in `AppColors.expense` (red) with a minus sign.

### Balance Calculation

- **Account balance**: For checking accounts, the balance field is updated on every transaction add/edit/delete.
- **Credit card balance**: Represents the current bill amount. `creditLimit - balance = available credit`.
- **Total balance**: Sum of all active checking account balances minus credit card bills.
- **Monthly summary**: Aggregated from transactions filtered by current month.
  - `totalIncome` = sum of all income transactions in the period.
  - `totalExpenses` = sum of all expense transactions in the period.
  - `netResult` = `totalIncome - totalExpenses`.

### Reconciliation

- Each transaction has an `isReconciled` flag (default `false`).
- Users can mark transactions as reconciled (confirmed against bank statement).
- Reconciliation status is displayed visually (checkmark icon, muted style for reconciled items).
- Account detail page shows reconciled vs. unreconciled totals.

## Caching & Local Storage

### In-Memory Cache (`AppDataCache`)

- Located at `core/cache/app_data_cache.dart`.
- Stores: `accounts`, `categories`, `transactions` (current month), `currentUser`, `dashboardSummary`.
- **Not persisted to disk** — cleared when the app is killed.
- Repositories check cache before hitting Firestore; write results back to cache.
- `forceRefresh` parameter on repository methods bypasses cache (used by pull-to-refresh).
- `clear()` is called on sign-out to reset all cached data.
- `isFullyLoaded` convenience getter: `true` when accounts + categories + current month transactions are cached.

### Local Preferences (`SharedPreferences`)

- Theme mode preference (light / dark / system).
- Last selected period filter.
- Onboarding completed flag.

### Startup Flow (`StartupCubit`)

```
App Launch → /startup → Wait for AuthBloc to resolve
  ├─ Unauthenticated → Navigate to /onboarding
  └─ Authenticated → Load accounts → Load categories → Load transactions (current month)
                    → Load dashboard summary → Populate singleton blocs → Navigate to /
```

- `StartupCubit` listens to `AuthBloc.stream` to wait for auth resolution.
- Reports progress via `StartupLoading(step, progress)` for animated UI.
- On error, emits `StartupError` with retry support.

## Design Principles

- **SOLID** — Single responsibility per class; depend on abstractions.
- **DRY** — Extract shared logic into `core/utils` or shared widgets.
- **KISS** — Favour simplicity. Don't add abstractions unless they reduce complexity.
- **Composition over inheritance** — Prefer composing small widgets.
- **Responsiveness** — Every page must render well on mobile, tablet, and desktop.
- **Performance** — Use `const` constructors, cached images, shimmer loading.

## Coding Conventions

| Rule | Detail |
|---|---|
| **Naming** | `snake_case` files, `PascalCase` classes, `camelCase` members |
| **Imports** | Use package imports (`package:financo/...`), not relative |
| **`const`** | Apply `const` constructors and `const` values wherever possible |
| **Private fields** | Prefix with `_`; expose only what's needed |
| **File length** | Keep files under 200 lines; split if larger |
| **Widget decomposition** | Extract widgets when they exceed ~50 lines or are reused |
| **Comments** | Only where the *why* isn't obvious; avoid restating the code |
| **Lint rules** | Follow `very_good_analysis`; never add `// ignore` without justification |
| **Error handling** | Always use `Either<Failure, T>` in use cases; never throw from repositories |
| **Bloc naming** | Events: past tense (`TransactionsLoaded`); States: noun + status (`DashboardState`) |
| **i18n** | All strings via `t.section.key` — never hardcode user-facing text |
| **Currency** | Always format with `formatCurrency()` — never display raw doubles |

## When Adding a New Feature

1. Create the feature folder under `lib/features/` with `data/`, `domain/`, `presentation/` subfolders.
2. Define entities in `domain/entities/` — pure Dart classes.
3. Define the repository contract in `domain/repositories/`.
4. Create use cases in `domain/usecases/` — one per business action.
5. Implement models in `data/models/` with `fromJson`/`toJson`.
6. Implement data sources in `data/datasources/`.
7. Implement the repository in `data/repositories/`.
8. Create Bloc/Cubit in `presentation/bloc/` or `presentation/cubit/`.
9. Build pages in `presentation/pages/` and widgets in `presentation/widgets/`.
10. Register all dependencies in `app/di/injection_container.dart`.
11. Add routes in `app/routes/app_router.dart`.
12. Add all user-facing strings to `en.i18n.json` and run `dart run slang`.

## Post-Implementation Checklist

After every code change (new features, bug fixes, refactors), **always**:

1. Run `dart run slang` if any i18n JSON file was modified.
2. Run `flutter analyze` and verify **zero** errors, warnings, **and** info-level issues.
3. Fix every reported issue — this project uses `very_good_analysis` (strict):
   - `prefer_const_constructors`, `lines_longer_than_80_chars`, `directives_ordering`, `use_build_context_synchronously`, `deprecated_member_use`, `discarded_futures`, etc.
4. Never add `// ignore` without clear justification in a comment.
5. Treat info-level lint hints the same as warnings — they must be resolved.
